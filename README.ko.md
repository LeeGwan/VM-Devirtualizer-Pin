# VMAnalyzer — Pin으로 만든 VM 난독화 분석 도구

> [English](./vmanalyzer-post.md) | 한국어

VM 기반 난독화(가상화 보호)를 분석해 보려고 Intel Pin으로 만든 동적 분석
툴입니다. VM 디스패처를 런타임에 후킹해서, 핸들러가 무엇을 읽고 쓰는지를
영역 라벨이 붙은 로그로 뽑아줍니다.

## 가상화 난독화가 왜 까다로울까요

가상화 코드는 원본 코드를 숨기는 게 아니라, 코드를 가상화해서 실제 CPU가 직접
처리하지 않게 만듭니다. 대신 가상 CPU를 만들어 그 코드를 처리하게 하죠. 그 안에서는
실제 CPU에서 가상 CPU로 넘어가기 전에 컨트롤러가 레지스터 값들과 여러 준비를 한 뒤,
fetch instruction에서 opcode를 읽어오고, 암호화된 것을 복호화한 다음, 그에 맞는
`vm_handler` 몇 번째 핸들러를 호출합니다.

하지만 거기까지 가는 길에는 난독화된 코드나, 전혀 필요 없는 가비지 코드 또는 가비지
핸들러가 수없이 많습니다. 그래서 코드 가상화가 적용돼 있으면 리버싱이 너무 힘들어지고
시간도 오래 걸리죠. 그걸 줄여보려고, Intel Pin API를 사용해서 어떤 핸들러가 도는지,
무엇을 읽고 쓰는지를 보고 필요 없는 코드는 하나하나 필터링하면서, 진짜 일하는 핸들러와
코드만 추출해 분석할 때 도움을 주는 도구를 만들었습니다.

## 만들면서 겪은 것들

처음 버전은 그냥 VM 영역 안의 모든 메모리 읽기/쓰기를 통째로 덤프했습니다.
동작은 했지만 로그가 **쓰레기 값**으로 도배가 되더라고요. 디스패처 자신의
디코드 루틴이 메모리를 끊임없이 건드리는데, 그게 전부 노이즈로 섞여
들어왔거든요.

해결은 Pin API를 좀 더 파고들면서 풀렸습니다. 스택 접근을 정확히 가려내는
`INS_IsStackRead`·`INS_IsStackWrite`, 핸들러로 점프하는 순간을 잡아주는
`IPOINT_TAKEN_BRANCH`, 그리고 디코드 루틴이 스택에 쓰는 시그니처 값(`0x4b82`,
`0x25c5`)으로 노이즈 구간을 감지해 다음 핸들러에 진입할 때까지 그 구간의 로그
출력을 건너뛰는 처리 — 이런 API들을 하나씩 찾아 붙여가면서 트레이스를 점점
다듬었고, 결국 원하는 만큼만 깔끔하게 결과가 나오도록 만들 수 있었습니다.

## 동작은 이렇습니다

- `.v-lizer` 섹션을 찾아 VM 영역을 잡습니다.
- `PUSHAD`/`POPAD`를 기준으로 가상화 실행 구간의 추적을 켜고 끕니다.
- 디스패처 점프(`jmp [edi+eax*4]`)에서 핸들러 진입과 가상 opcode를 기록합니다.
- 메모리 읽기/쓰기와 `LEA`를 잡아서, 주소와 값을 영역(`vm_context`, `[STACK]`,
  `[MEM]` 등) 라벨과 함께 출력합니다.
- 종료할 때 핸들러별 호출/읽기/쓰기 횟수를 요약해 줍니다.

위쪽 트레이스를 보면 한 핸들러(`op=0x1a`)가 스택과 `init_flag`를 어떻게 읽고
쓰는지가 그대로 드러나고, 아래 요약표는 핸들러별 호출/읽기/쓰기 횟수와 디코딩된
opcode를 한눈에 보여줍니다(값은 전부 16진수). 각 핸들러의 의미를 하나씩 복원해
나가는 출발점으로 쓰기 좋습니다.

> 참고: 요약표의 reads/writes는 디코딩 노이즈까지 포함한 raw 카운트라, 위 트레이스에
> 실제로 찍힌 줄 수보다 클 수 있습니다.
